# DrawCall 代码绘制优化设计

## 目标

在不改变玩法、不把麻将牌整体替换成图片资源的前提下，减少运行时独立 `Graphics` 组件和 Graphics/Label 交错渲染造成的 DrawCall。

## 方案

### 棋盘

- `BoardLayer` 保留牌节点作为触摸和动画载体，但牌节点不再各自持有背景、牌面 `Graphics`。
- 新增公共棋盘绘制节点和单一 `Graphics`，由 `redrawBoardGraphics()` 按牌节点的实时位置、旋转、缩放和高亮状态统一绘制全部牌面。
- 麻将字符与数字保留为 Label，但移动到独立文字层，并同步牌节点的变换，以保证所有 Graphics 先绘制、所有 Label 后绘制。
- 牌的创建、拖动、回滚、高亮、消除和恢复入口统一触发公共棋盘重绘，不改变棋盘模型与触摸判定。

### 特效

- 消除碎片不再创建 24 个独立 `Graphics`，改为公共特效 `Graphics` 加轻量状态数组，每帧统一重绘。
- 闪光与圆环优先并入公共特效绘制；浮动分数继续使用 Label，以控制改动范围。
- 提示框可以继续保留现有交互节点，但其代码绘制逐步汇入公共特效 Graphics，确保提示行为不变。

### UI 与计时

- 本轮不重构首页、设置和结算界面的全部节点结构，避免一次性改动过大。
- 计时 Label 只在显示秒数变化时更新，减少文字重建。
- 不引入新图片、BMFont、自定义材质或玩法改动。

## 验证

- 添加静态回归检查：牌节点不再创建自己的 `Graphics`，棋盘存在唯一公共绘制入口，文字节点位于独立层，碎片使用公共特效状态而不是逐碎片 Graphics。
- 运行现有 `scripts/check-duidui-regressions.js`。
- 运行 TypeScript 编译检查。
- 在 Creator 性能面板中人工确认 DrawCall；静态测试只能验证结构，不能替代真机/预览性能数据。

## 完成标准

- 玩法、拖动、高亮、提示、消除和恢复行为保持一致。
- 常驻棋盘 Graphics 组件数不再随牌数线性增长。
- 消除碎片 Graphics 组件数不再随粒子数增长。
- 所有现有回归检查和 TypeScript 检查通过。
